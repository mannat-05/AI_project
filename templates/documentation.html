1. How the Model Works

This project uses a hybrid search strategy combining an A* style pathfinder with a custom ElasticHashSet to efficiently track visited states.

  • ElasticHashSet: A multi-layer open-addressing hash set inspired by elastic hashing. It stores state entries across three shrinking layers to reduce collision cost and speed up membership checks.
  • Hybrid Elastic A*: Extends classical A* by using 8-way movement and a turn penalty. The solver can move diagonally, but diagonal progress is blocked when adjacent cardinal neighbors are walls.
  • Octile heuristic: Estimates travel cost for diagonal grids while preserving admissibility. This is ideal for 8-way grid motion.
  • Turn penalty: A small extra cost for changing direction encourages smoother paths and better real-world motion profiles.

2. Search Model Integration

The web simulator integrates the same search logic from the project environment:

  • The backend uses web/solver.py, which mirrors the same algorithm from master_run.py.
  • The search solver receives a generated maze and returns a complete path from the start at (0, 0) to the goal at the bottom-right cell.
  • The returned path is rendered in the browser, while metrics are reported back to the user in real time.

3. Simulation and Maze Environment

The browser simulation reuses the same maze concepts found in the Python environment:

  • Random scattered maze: A dense obstacle field with guaranteed solvability.
  • Structured corridor maze: A carved passage layout resembling competition-style maze environments.
  • The grid display highlights walls, free space, the start cell, the goal cell, and the computed path.

The simulator uses a canvas to draw the maze and animates the path as a green trail.

4. Architecture

Files added for the website:

  • web_app.py: Flask backend with API routes for solving mazes.
  • web/solver.py: Search algorithm and maze factory logic.
  • templates/index.html and templates/documentation.html: Frontend pages.
  • static/app.js and static/style.css: Browser rendering and styling.

This architecture makes the site easy to extend while preserving the core solver used by the existing Python project.
